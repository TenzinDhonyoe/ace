## Vision

### The 10x Check

The original plan treats the exam-review site as the product and a generator as plumbing. That framing caps the ceiling at "a tool for upper-year eng students to cram before finals" — a real but bounded market.

The 10x reframe: **the product is the library of AI-composable interactive visualizations. The generator is the reference application that proves the library.** Same code this weekend. 10x the long-term ceiling.

Why 10x is the right number here:
- Audience expands from "students who want a study site" to "any AI tutor, ed-tech startup, or interactive textbook publisher." That's ~100x wider.
- Moat shifts from "we have a nice template" (weak, copy-paste-able in a day) to "we have the component library everyone else depends on" (strong, network-effects via dependency graph).
- Distribution shifts from "students find the GitHub repo" (hard) to "students find the Claude skill + developers find the npm package" (two orthogonal loops).

### The Platonic Ideal

In 12 months, if this works:

A biomedical engineering student at McGill opens Claude.ai, says "help me review for BIOE 310, here are the PDFs," and in 90 seconds has a hosted interactive review page with live protein-folding sandboxes generated from her lecture content. She doesn't know there's a component library behind it.

A two-person ed-tech startup in Seoul building an AI tutor for Korean high schoolers prepping for suneung imports `@ace/components` and skips 4 months of visualization engineering. Their tutor now has live physics simulations their competitors don't.

A physics professor at Waterloo runs the `ace-template` against his course PDFs, publishes the output as supplementary material, and his students use it more than the textbook.

All three use the same 30 components. That library is the product.

### North Star

**"The shadcn/ui of educational interactives."** Open-source, composable, AI-generable, universally usable. The exam-review generator is the flagship demo that proves the pattern.

## Accepted Scope (now in Phase 1)

- Monorepo structure (`ace-components`, `ace-template`, `ace-skill`, `gallery`)
- Five widget primitives extracted as standalone components
- Versioned `exam-structure.md` schema v0.1
- localStorage persistence for all widgets
- Parallel Claude skill distribution
- Gallery landing page with live widget embeds
- Infrastructure-first framing in all launch materials

## Deferred to TODOS

- VS Code extension consuming `ace-components` (Phase 3+)
- Professor-facing workflow ("upload syllabus, generate full course materials") (Phase 3+)
- Additional widget types beyond the initial 5 (add as real courses demand them, not speculatively)

## NOT in Scope (explicitly)

- Accounts / auth (never, in the open-source version)
- Database / server state for Phase 1 (localStorage only)
- Payments in Phase 1 (hosted tier only, and only after Phase 1 validates)
- Mobile native apps (web is enough; the sandboxes work on mobile browsers)
- Competing with Quizlet / Chegg on their terms (flashcard decks, answer banks) — we compete on interactive visualization, not the commodity features

## Temporal Interrogation — Decisions to Resolve Now

**Hour 1 (foundations):**
- Monorepo tool: pnpm workspaces or bun workspaces? *Decision needed.* Recommend bun (faster, simpler, matches the vibe).
- Component framework: React or vanilla JS + Web Components? BME804 appears to be vanilla. *Decision needed.* Recommend vanilla-JS + light wrapper for React interop — keeps the library framework-agnostic, which matters for "infrastructure" framing.
- License: MIT. Confirmed in office-hours.

**Hour 2-3 (core logic):**
- How does a widget declare its props schema? JSON Schema? TypeScript? This affects AI generation quality. *Decision needed.* Recommend JSON Schema embedded per component — machine-readable, language-agnostic, Claude-friendly.
- How does `ace-template` discover available widgets from `ace-components`? Static import list vs. registry pattern. Recommend registry pattern from day 1 — cheap now, expensive to retrofit.

**Hour 4-5 (integration):**
- How does the generator prompt know which widget to use for which content? Heuristics in `CLAUDE.md` + the schema itself. Invest real time here — this is the quality-lives-or-dies step.

**Hour 6+ (polish/tests):**
- Visual regression testing for widgets — skip for Phase 1, add when a widget breaks in production.
- Accessibility — tab navigation + ARIA labels in every widget from day 1. Not after.

## Why This Plan Will Probably Work

- **Real demand is validated.** 5 classmates return across multiple courses. That's behavior, not interest.
- **The moat is real.** Nobody auto-generates working physics sandboxes from PDFs today. Whitespace.
- **The reframing is free.** Moving from "study site generator" to "infrastructure" costs zero code and roughly 10xes the ceiling.
- **The build is small.** 2 weekends with CC. Even if the library thesis is wrong, you ship a working generator and have lost nothing.
- **Distribution is layered.** Dev audience via npm + GitHub. Student audience via Claude skill. No single-point-of-distribution dependency.

## Why This Plan Might Not Work (honest failure modes)

- Generator output quality on courses far from engineering (e.g., literature, history) could be poor. Mitigation: gallery lets you show only the cases it works well on; schema version can grow to add non-numeric widget types.
- Claude skill distribution may not reach students at scale in Phase 1 timeframe. Mitigation: template repo + gallery are independent of this.
- Library adoption by other devs is slow and uncertain. Mitigation: even zero external adoption still leaves a working generator + solid portfolio project.
- You ship the components but the generator's prompt engineering is weaker than hand-tuning. Mitigation: the `CLAUDE.md` is the hardest part and gets most of your Phase 1 attention.
