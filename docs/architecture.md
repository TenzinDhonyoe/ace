# Design: Ace — Open-Source Infrastructure for AI-Generated Interactive Study Content

Status: APPROVED

Reference artifact: https://bme804-review.vercel.app/#act

**North Star (12-month ideal):** The shadcn/ui of educational interactives. An open-source library of AI-composable interactive physics/math/bio visualization components that any AI tutor, study tool, or ed-tech app can pull from. The exam-review site generator is the flagship reference implementation that proves the library works.

**Reframed positioning:** This is NOT "a study site generator." It is **open-source infrastructure for AI tutors**, with a flagship generator as the demo. Same code, 10x the ceiling. Audience expands from students only → students + every ed-tech developer shipping AI features.

## Problem Statement

Upper-year engineering students studying for concept-visualization-heavy finals (problem sets + MCQs) currently re-read lecture PDFs passively and ask ChatGPT to "quiz me." Neither approach produces the interactive, visual understanding the material demands. The creator built a hand-tuned interactive study site (BME804) — live SVG sandboxes driven by sliders, per-section flashcards, self-test quiz — and 5 classmates returned to it across multiple courses. The goal is to turn this one-off artifact into a reproducible generator anyone can use.

## Demand Evidence

- 5 classmates used the BME804 site and **came back asking for sites for other courses** (the strongest signal in the session — repeat usage across courses, not one-exam novelty)
- External signal: Jibraan Kadri (uofg / uoft) publicly commented suggesting the creator wrap and ship it
- Unknown whether the site spread outside the creator's school (could be upside)
- Competing behavior replaced: lecture re-reading + ChatGPT quiz-me. Neither produces interactive visualizations.

## Status Quo

Students currently study by: (1) re-reading lecture PDFs (passive, low retention), (2) asking ChatGPT to generate MCQs (no interactivity, no persistent structure, no physics sandboxes), (3) ad-hoc Quizlet decks (flashcards only, no computation). Cost: hours of low-retention studying, suboptimal exam performance, real GPA/scholarship/grad-school consequences.

## Target User & Narrowest Wedge

**Target user:** Upper-year undergrad engineering student (years 3–4) facing a concept-visualization-heavy final with MCQ format, where a poor grade meaningfully hurts GPA, scholarship standing, or grad-school prospects.

**Narrowest wedge:** Student forks a GitHub template, drops lecture PDFs into `/inputs/`, writes a short `exam-structure.md`, runs Claude Code with a provided slash command, deploys the output to Vercel. Zero hosting cost to the creator. Ships this weekend.

## Constraints

- Side project — cannot become an ops job with a credit card bill
- Open source first — maximum reach prioritized over revenue
- Phase 2 (hosted) may charge /site to capture non-technical users
- Quality bar = the BME804 site (hand-tuned). Must not ship a 70%-as-good generator that destroys the brand before it's built.

## Premises (all agreed)

1. The moat is the **interactive SVG sandboxes**, not the quizzes. ChatGPT can quiz; it cannot auto-generate working physics visualizations wired to sliders. That's the demo-able magic.
2. Output = a single-page deployed static site. No login, no accounts, no database. The student owns the HTML.
3. We are NOT building a study platform. We are building a generator. The output is the product; the generator is plumbing.
4. Distribution is the hard part. With Claude + a decent template, generation is a weekend. Getting 1,000 students to use it is the real work.
5. MVP = template repo + Claude Code slash command. Hosted free on Vercel by the user.
6. Creator keeps hand-making sites for their own courses — these are the content flywheel, marketing proof, and generator QA all at once.

## Approaches Considered (post-CEO-review)

### Approach A (ORIGINAL): Template repo only
Rejected as sole Phase 1 vehicle — caps audience at Claude Code users (~50k), excludes the stated target user (non-technical upper-year eng students), and structures the codebase around the incidental artifact (template) rather than the valuable one (components).

### Approach B (CHOSEN for Phase 1): Component library + template consumer
Monorepo: `/packages/ace-components` (the reusable SVG sandbox library — sliders, physics widgets, flashcards, self-test) + `/packages/ace-template` (reference consumer that generates the full review site). Ships as two npm packages. The template is one of several future vehicles that consume the library.
- **Effort:** S-M (~2 weekends; +1 weekend vs. template-only). CC-compressed: ~1 extra afternoon.
- **Risk:** Low. If nobody uses the library standalone, it's just well-organized code.
- **Unlocks:** Every future distribution vehicle (hosted app, Claude skill, VS Code extension, other devs' apps) pulls from the same library. No rewrite needed later.

### Approach C: CLI tool (`npx ace`) — REJECTED
Requires user-owned Anthropic API key. Too much friction for students.

## Accepted Scope Expansions (from CEO review)

1. **Component library + template consumer structure (Proposal 1, Approach B).** Monorepo. Components shippable standalone.
2. **Claude skill as parallel Phase 1 vehicle (Proposal 2).** `ace-skill` — ~100 lines of markdown + reference to the component library. Lets non-technical students use it today inside Claude.ai. Removes urgency of Phase 2 hosted app.
3. **Public gallery landing page (Proposal 3).** Static Next.js site showing live slider thumbnails of every course site (yours + early users'). Every new site = viral surface area. Seeds future UGC library.
4. **Versioned `exam-structure.md` schema (Proposal 4).** Start with 5 widget types that exist in BME804 today (slider-sandbox, flashcard, MCQ, diagram-label, concept-card). Resist over-prescription. Document the schema; version it. This is where generator consistency lives or dies.
5. **localStorage persistence for "Got it" / "Review" buttons (Proposal 5).** Widget-level. No backend. Turns one-night cram tool into multi-week spaced-review companion.
6. **Reframe as "open-source infrastructure for AI tutors" (Proposal 6).** README, launch post, landing page copy all lead with the library and position the generator as the reference app. Free repositioning; changes who stars, writes about, and builds on this.

## Resolved Architectural Decisions (from /plan-eng-review)

1. **Package manager:** bun workspaces. One tool, faster install, built-in test runner. Low blast radius for a 4-package monorepo with zero exotic tooling.
2. **Component model:** plain-JS classes with `mount(el, props)` / `destroy()` API. NOT React, NOT Web Components in Phase 1. Works in static HTML (primary target), React, Vue, everywhere. Web Components wrapper ships later as opt-in.
3. **Prop schemas:** JSON Schema per widget, co-located (`src/<widget>/schema.json`). Machine-readable by Claude without a compiler. Runtime-validated at mount. TS types auto-generated via `json-schema-to-typescript`.
4. **Widget discovery:** explicit `registry` object exported from `@ace/components` root. Adding a widget = one import + one key. Registry IS the API contract with Claude's generator prompt.
5. **CSS isolation:** BEM-prefixed class names (`.ace-slider-sandbox__track`). No Shadow DOM in Phase 1. Grep-able, restyleable by site authors.
6. **Bundle format:** ESM primary, one UMD IIFE per widget for `<script>`-tag consumption on static HTML.
7. **Site config:** generator emits `site.config.json` (decoupled from HTML). Template renders the config. Swappable generators later (GPT, Gemini) without touching template.
8. **PDF parsing:** rely on Claude Code's native PDF ingestion. Validate with a 20-page lecture in Weekend 1. Fallback = pre-process with `pdftotext` to markdown.
9. **Bundle budget:** ≤15KB gzipped core, ≤5KB per widget. Enforced via `size-limit` in CI.
10. **Error boundaries:** every widget's `mount()` is try/catch-wrapped. Failures render an inline error card; no white-screen.
11. **localStorage namespacing:** `ace:${siteId}:${widgetId}:${key}`. Injected `siteId` at generation time. Avoids cross-site collisions on shared domains.
12. **License:** MIT. Maximum reach.
13. **Canonical showcase:** BME804 regenerated from the new pipeline + 1 additional course in a different domain. Both embedded live on the gallery landing page.

## Next Steps (Build Order — revised)

### Weekend 1 — Library foundations

**Lane A (sequential — shared `packages/ace-components/` dir):**
1. Monorepo setup with bun workspaces. Four packages: `@ace/components`, `@ace/template`, `ace-skill/`, `apps/gallery`. Root `package.json` + `bunfig.toml` + GH Actions scaffold.
2. Write `schema.json` v0.1 for `site.config.json` (the config the generator emits). This is the contract everything else depends on. Do this BEFORE any widget code.
3. Extract 5 widget primitives from BME804 as plain-JS classes with `mount(el, props)` / `destroy()` / `static schema`:
   - `SliderSandbox`, `Flashcard`, `MCQQuiz`, `DiagramLabel`, `ConceptCard`
   - Each: `src/<widget>/index.js` + `schema.json` + `examples.json` + `styles.css` + `README.md` + `index.test.js`
4. `lib/storage.js` helper: namespaced `localStorage` with graceful fallback when unavailable. Used by every widget.
5. Unit tests (bun test + jsdom): happy render, bad-props rejection, destroy cleanup, storage roundtrip, storage-unavailable fallback. Target: all widgets covered.

**Lane B (independent — parallel with Lane A):**
6. `eval/` harness: 3 golden BME804-style PDFs + expected widget-type distribution JSON. `bun run eval` script (stub — full implementation wires up in Weekend 2 once CLAUDE.md exists).
7. `size-limit` config enforcing ≤15KB core + ≤5KB per widget.

### Weekend 2 — Generator, skill, gallery, launch

**Sequential (depends on Weekend 1):**
8. Write `CLAUDE.md` with the 7-section structure (Role, Input, Output, Widget Catalog, Quality Rules, Process, Self-Check). Include few-shot examples per widget. **This is the highest-leverage artifact in the project — invest half the weekend here.**
9. Write `/generate-review-site` slash command that orchestrates: read `inputs/`, emit `site.config.json`, validate against schema, render via `@ace/template` to `/output/`.
10. `@ace/template` renderer: reads `site.config.json`, mounts widgets from `registry`, wraps each in an error boundary card, emits single-file HTML + CSS bundle.
11. **Dogfood:** regenerate BME804 from scratch via the new pipeline. Compare to the reference site. Iterate on CLAUDE.md until output quality matches. Then generate a second course in a different domain (recommend: a physics or circuits course you haven't hand-tuned).
12. `ace-skill/` — Claude skill file (markdown + slash command) for non-technical students via Claude.ai.
13. `apps/gallery` — Next.js landing page with live widget embeds (not screenshots) + links to every generated course site. Framing: "infrastructure for AI tutors."
14. CI pipeline: PR runs tests + `size-limit`; git tag publishes `@ace/components` and `@ace/template` to npm via `changesets`.
15. **Launch:** Show HN, Twitter/X demo clip of a live slider, r/EngineeringStudents, ed-tech newsletters, reply to @KadriJibraan.
